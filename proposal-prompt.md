## 0.5. 사전 브리핑 (에이전트 작업 전 필수)

> **입력을 분석한 직후, 에이전트를 투입하기 전에** 아래 프로세스를 수행합니다.
> **핵심 원칙: 에이전트가 먼저 분석하고 추천 → 사용자가 결정**

### STEP A: 기술 스택 추천/개선

입력에 기술 스택이 **지정되지 않은 경우:**
- 프로젝트 특성에 맞는 스택 2~3개를 추천 + 추천 사유를 설명합니다.

**추천안 예시 (프로젝트별로 달라짐):**
| 추천 | 스택 | 장점 | 적합한 상황 |
|------|------|------|------------|
| A (추천) | Next.js + Supabase | 풀스택 빠른 개발, SSR/SSG | 웹 중심, MVP, 빠른 출시 |
| B | React + Node.js + PostgreSQL | 높은 자유도, 대규모 확장 | 복잡한 백엔드, 대규모 서비스 |
| C | Flutter + Firebase | 크로스플랫폼, 실시간 | 앱 중심, 실시간 기능 |

입력에 기술 스택이 **지정된 경우:**
- "이 스택도 좋지만, [대안]이 더 나을 수 있다" 형태로 대안 제안 + 비교표를 보여줍니다.
- 사용자가 최종 선택합니다.

### STEP B: 기능 개선/추가 제안

입력된 기능 목록을 분석하여:

1. **개선 제안** — UX/성능/보안 관점에서 기존 기능의 개선안 제시
2. **추가 제안** — 유사 서비스 기준으로 빠진 기능을 3단계로 분류하여 제안

| 구분 | 설명 | 예시 |
|------|------|------|
| 필수 | 없으면 서비스 출시 불가 | 로그인, 결제, 알림 |
| 추천 | 경쟁력/UX에 큰 영향 | 실시간 채팅, 대시보드, 검색 필터 |
| 선택 | 차별화/부가가치 | AI 추천, 다국어, 게이미피케이션 |

- 사용자가 최종 선택합니다.

### STEP C: 나머지 확인 사항 (AskUserQuestion 도구 사용)

| # | 질문 | 추천안 예시 |
|---|------|-----------|
| Q1 | 프로젝트 유형/플랫폼은? | 반응형 웹 / 웹+앱 / 하이브리드 / SPA |
| Q2 | 프로젝트 규모/일정은? | 소(5~10화면, 1~2개월) / 중(15~30화면, 3~4개월) / 대(30+화면, 5개월+) |
| Q3 | 예산 범위는? (선택 가능) | 미지정 시 M/M 기반 자동 산출 |
| Q4 | 제안서 톤 & 강조점은? | 기술형 / 비즈니스 가치형 / 밸런스형 |

> Q1~Q4 각 질문마다 프로젝트 특성에 맞는 **추천안을 먼저 제시**하고, 사용자가 선택하거나 직접 입력합니다.
> 미팅 요구사항에서 이미 추출 가능한 항목은 건너뜁니다.

### 브리핑 결과 → 에이전트에 전달

사용자 답변을 정리하여 **모든 에이전트 프롬프트에 아래 블록을 추가**합니다:

```
## 사전 브리핑 (사용자 지정)

- 프로젝트 유형: [선택한 유형/플랫폼]
- 기술 스택: [최종 선택한 스택]
- 핵심 기능: [확정된 기능 목록 — 필수/추천/선택 구분]
- 규모/일정: [선택한 규모]
- 예산 범위: [지정 또는 "M/M 기반 자동 산출"]
- 제안서 톤: [선택한 톤]
- 추가 요구사항: [기타 사용자 지정 사항]

⚠️ 위 브리핑 내용을 반드시 반영하여 작업하세요.
```

## 1. 팀 아키텍처

### 에이전트 구성 (4명 + 팀 리드)

```
┌──────────────────────────────────────────────────┐
│              team-lead (팀 리드)                    │
│    태스크 생성, 의존성 관리, 최종 통합, 검수           │
├──────────────────────────────────────────────────┤
│                                                    │
│  [1] analyst    → 요구분석 + 기술설계 전문가          │
│  [2] planner    → 일정/인력/예산 전문가    ← 병렬     │
│  [3] writer     → 본문/UI·UX/회사소개 작성  ← 병렬    │
│  [4] reviewer   → 교차 검수 전문가                   │
│                                                    │
└──────────────────────────────────────────────────┘
```

### 태스크 의존성 & 병렬 처리 구조

```
[Task #1] 요구분석 + 기술설계 (analyst)
    │
    ├──────────────────┐
    v                  v
[Task #2] 일정/인력   [Task #3] 본문 작성        ← #1 완료 후 동시 병렬 실행
 /예산 (planner)      (writer)
    │                  │
    └──────┬───────────┘
           v
[Task #4] 교차 검수 (reviewer)                   ← #2, #3 모두 완료 후
           │
           v
[Task #5] 최종 통합 (team-lead)                  ← #4 완료 후
```

## 2. 팀 생성 & 태스크 설정 절차

### STEP 1: 팀 생성

```
TeamCreate:
  team_name: "proposal-team"
  description: "웹/앱 개발 제안서 작성 AI 드림팀"
```

### STEP 2: 프로젝트별 output 디렉토리 자동 생성

```bash
PROJECT_DIR="output/proposal/$(date +%Y%m%d)_[프로젝트명]"
mkdir -p "$PROJECT_DIR/.work"
```

**폴더 구조:**
```
output/proposal/[YYYYMMDD]_[프로젝트명]/
├── final_proposal.md                    ← 최종 제안서 (사용자용)
└── .work/                               ← 중간 파일 (에이전트 작업물)
    ├── 01_analysis_and_architecture.md  (analyst)
    ├── 02_plan_and_budget.md            (planner)
    ├── 03_proposal_content.md           (writer)
    └── 04_review_report.md              (reviewer)
```

### STEP 3: 태스크 5개 생성 + 의존성 설정

| Task ID | Subject | Owner | Blocked By | Output File |
|---------|---------|-------|------------|-------------|
| #1 | 요구분석 + 기술설계 | analyst | - | `{PROJECT_DIR}/.work/01_analysis_and_architecture.md` |
| #2 | 일정/인력/예산 산출 | planner | #1 | `{PROJECT_DIR}/.work/02_plan_and_budget.md` |
| #3 | 제안서 본문 작성 | writer | #1 | `{PROJECT_DIR}/.work/03_proposal_content.md` |
| #4 | 교차 검수 | reviewer | #2, #3 | `{PROJECT_DIR}/.work/04_review_report.md` |
| #5 | 최종 통합 제안서 | team-lead | #4 | `{PROJECT_DIR}/final_proposal.md` |

### STEP 4: 에이전트 순차/병렬 투입

```
1단계: analyst 투입 → Task #1 완료 대기
2단계: planner + writer 동시 투입 (병렬) → Task #2, #3 모두 완료 대기
3단계: reviewer 투입 → Task #4 완료 대기
4단계: 팀 리드가 직접 Task #5 수행 (최종 통합)
```

### STEP 5: 정리 & 팀 종료

```
1. 전체 에이전트에 shutdown_request 전송
2. TeamDelete로 팀 정리
3. 사용자에게 안내:
   - output/proposal/[날짜]_[프로젝트명]/final_proposal.md (최종 제안서)
   - .work/ 폴더에 중간 작업물 보존됨
```
